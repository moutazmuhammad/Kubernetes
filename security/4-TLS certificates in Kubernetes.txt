- تتكون مجموعة Kubernetes من مجموعة من العقد الرئيسية master والعقد العاملة worker.

- بالطبع ، يجب أن تكون جميع الاتصالات بين هذه العقد آمنة ويجب تشفيرها. يجب أن تكون جميع التفاعلات بين جميع الخدمات وعملائها آمنة.

	* على سبيل المثال ، يجب على المسؤول الذي يتفاعل مع مجموعة Kubernetes من خلال أداة التحكم kubectl أو من خلال الوصول إلى Kubernetes API مباشرة إنشاء اتصال TLS آمن.

- يجب أيضًا تأمين الاتصال بين جميع المكونات داخل مجموعة الفيروسات التاجية.

- لذا فإن المطلبين الأساسيين هما أن تكون جميع الخدمات المختلفة داخل cluster لاستخدام شهادات الخادم وجميع العملاء لاستخدام شهادات العميل للتحقق من هويتهم.

-  نظرة على المكونات المختلفة داخل مجموعة Kubernetes وتحديدا الخوادم:
	1- خادم api (api server)
		
		* يعرض خادم API: و HTTP كخدمة تستخدمها المكونات الأخرى بالإضافة إلى المستخدمين الخارجيين لإدارة مجموعة Kubernetes.
		* لذلك فهو خادم ويتطلب شهادات لتأمين جميع الاتصالات مع عملائه.
		* لذلك نقوم بإنشاء شهادة ونحتفظ بها هنا. نحن نسميها apiserver.crt و apiserver.key
	
	2- خادم atcd server
		* يخزن خادم Etsy جميع المعلومات حول cluster ، لذلك يتطلب زوجًا من الشهادات والمفتاح etcdserver.crt و etcdserver.ket
	
	3- خادم kubelete server
		*  تعرض أيضًا نقطة نهاية HTTP API التي يتحدث api server معها للتفاعل مع العقد العاملة worker.
		* يتطلب ذلك شهادة ومفتاح kubelete.crt و kubelete.key

- نظرة على مكونات العملاء:
	1- العملاء الذين يصلون إلى خادم API هم admins في الذين يستخدمون kubectl أو API.
	    يتطلب المستخدم الإداري شهادة ومفتاح للمصادقة على API server . ( admin.crt and admin.key)
	2- يتحدث المجدول إلى خادم API للبحث عن الأجزاء التي تتطلب جدولة ثم الحصول على API server  لجدولة الأجزاء الموجودة على عقد العمال الصحيحة.
		المجدول هو عميل يصل إلى  API خادم . (schedule.key and schedule.crt)
	3- مدير وحدة تحكم kube controler  هو عميل آخر يصل إلى API server  
	4-  اخر عميل kube proxy
